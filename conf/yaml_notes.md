## Building profile–dependent schema fields

The following table lists each schema mention that appears to depend on an external “building profile” dataset, along with its YAML line number and a suggested mitigation for working without that dataset.

| Mention text | YAML line number | Suggested mitigation |
| --- | --- | --- |
| `boilers: - id: string  # Must match building profile boiler tag` | `bms-snapshot.schema.yaml:L74–L76` | Treat `boilers[].id` as a free-form equipment identifier sourced from the snapshot or ingest (e.g., the BAS point group name, AHU/boiler tag parsed from trend filenames, or a stable hash/UUID). Drop the requirement that it match an external building profile; only require internal consistency across snapshots and analysis outputs. |
| `# OAT reset as currently configured in the BAS — compare against design values in the profile to detect drift, manual overrides, or misconfiguration.` | `bms-snapshot.schema.yaml:L85–L86` | Keep the `hws_oat_reset_active` fields as “current programmed values” only. If no design profile exists, set any unknown design targets to `null` and rely on heuristic checks (e.g., typical ranges, comparing to recent history, or analyst judgment) instead of strict profile-based comparisons. |
| `totals.electricity.pct_of_baseline: number  # Fraction of building's annual electricity baseline` | `analysis-output.schema.yaml:L115–L121` | Derive baseline electricity usage from the same inputs used to drive the analysis (e.g., a utility-bill time series, an externally provided annual kWh value, or roll-up from measured interval data). If no baseline is available, allow this field to be `null` and document the missing baseline in `open_questions`. |
| `totals.gas.pct_of_baseline: number` | `analysis-output.schema.yaml:L122–L127` | As above, compute gas baseline from external utility data or provided annual therm usage instead of a building profile record. If unavailable, permit `null` and surface a note that percent-of-baseline savings cannot be computed. |
| `totals.steam.pct_of_baseline: number` | `analysis-output.schema.yaml:L128–L133` | Same approach as other fuels: estimate steam baseline from utility statements or interval data where available; otherwise allow `null` and treat steam savings only in absolute terms (MLB, cost, CO₂e). |
| `totals.total.pct_cost_of_baseline: number` | `analysis-output.schema.yaml:L134–L138` | Compute total baseline cost as the sum of per-fuel baselines derived from rates and annual consumption, independent of any building profile record. If any fuel baseline is missing, either (a) compute percent-of-baseline from the subset with data, or (b) set this to `null` and flag the limitation. |
| `totals.total.pct_co2e_of_baseline: number` | `analysis-output.schema.yaml:L134–L138` | Compute baseline emissions from annual fuel usage and `factors.*.co2e_*` values without relying on a profile. If annual usage is unknown, allow `null` and use the absolute `co2e_tons_yr` savings only. |
| `# Projected compliance status after implementing ECMs — the analysis-specific counterpart to the current compliance status in the building profile.` | `analysis-output.schema.yaml:L157–L160` | Allow `regulatory_impact` to exist even when no formal building profile object is present by requiring the caller to supply the relevant baseline and cap values directly (from a separate compliance calculator, spreadsheet, or manual input). Treat this as “derived from external compliance inputs” rather than a strict building profile linkage. |
| `baseline_co2e_tons: number  # Current emissions (from building profile)` | `analysis-output.schema.yaml:L167–L168` | Populate `baseline_co2e_tons` from whatever emissions baseline is available (e.g., last full calendar year emissions computed from utility data) rather than a dedicated building profile. If no baseline exists, allow `null` and focus on directional changes (`co2e_reduction_tons`) computed from modeled savings. |
| `cap_co2e_tons: number  # Unchanged — restated here for readability` | `analysis-output.schema.yaml:L170` | Accept `cap_co2e_tons` as an explicit input tied to the jurisdiction/law (e.g., pulled from a separate policy ruleset or spreadsheet) instead of reading it from a building profile. If the cap is unknown, allow `null` and treat `projected_compliant`/`projected_penalty_usd_yr` as `null` or omitted. |
